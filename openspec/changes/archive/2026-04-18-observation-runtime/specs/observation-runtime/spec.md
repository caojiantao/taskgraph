# 运行时观测规范

## Purpose
定义 `taskgraph-kernel` 的标准运行时观测 SPI、事件模型、事件分发机制与关键触发时机，使调用方能够在不改变内核调度语义的前提下扩展日志、指标、Tracing 等外围可观测能力。

## ADDED Requirements
### Requirement: 内核必须提供标准运行时观测 SPI

`taskgraph-kernel` MUST 提供稳定的运行时观测扩展点，用于向外部暴露图执行与任务执行的关键事件。

#### Scenario: 调用方可按事件类型扩展观测逻辑

- **WHEN** 调用方希望监听某一类图或任务运行事件
- **THEN** 系统必须允许调用方按事件类型实现对应的观测处理器

#### Scenario: 观测 SPI 定义在内核中

- **WHEN** 框架提供运行时观测扩展能力
- **THEN** 该扩展点必须定义在 `taskgraph-kernel` 中
- **AND** 不应要求外围模块自定义另一套观测协议

### Requirement: 内核必须采用事件模型表达运行时观测

运行时观测 MUST 通过标准事件模型暴露，不得以单个超大监听器接口或线程池模板方法作为核心对外扩展机制。

#### Scenario: 观测事件至少区分图级与任务级

- **WHEN** 框架暴露运行时观测事件
- **THEN** 系统必须至少区分图级事件与任务级事件

#### Scenario: 不以线程池模板方法作为主扩展机制

- **WHEN** 框架提供 DAG 运行时观测能力
- **THEN** 系统不得以 `ThreadPoolExecutor.beforeExecute/afterExecute` 作为核心对外扩展模型

### Requirement: 内核必须提供事件分发能力

内核 MUST 提供统一事件分发器，用于把运行时事件分发给匹配的观测处理器。

#### Scenario: 分发器挂载在执行器上

- **WHEN** 调用方创建一个图执行器
- **THEN** 系统必须允许为该执行器配置一套运行时观测分发器
- **AND** 不应要求调用方在每次执行请求中重复传入分发器

#### Scenario: 未配置分发器时使用空实现

- **WHEN** 调用方未显式配置观测分发器
- **THEN** 系统必须提供默认的 no-op 分发器

#### Scenario: 父类事件处理器可以接收子类事件

- **WHEN** 某个处理器声明订阅某个事件基类
- **THEN** 该处理器必须能够接收该基类的全部子类事件

#### Scenario: 多个处理器按注册顺序同步接收事件

- **WHEN** 同一事件命中多个观测处理器
- **THEN** 系统必须在当前执行线程中按处理器注册顺序依次分发事件

### Requirement: 每次图执行必须生成唯一执行标识

内核 MUST 在每次图执行开始时生成唯一 `executionId`，并在该次执行的全部观测事件中复用该标识。

#### Scenario: 同一次图执行中的事件共享同一 executionId

- **WHEN** 同一张图的一次执行过程中发布多个运行时观测事件
- **THEN** 这些事件必须携带相同的 `executionId`

#### Scenario: 不同图执行的 executionId 必须不同

- **WHEN** 连续发起两次图执行
- **THEN** 两次执行产生的 `executionId` 必须不同

### Requirement: 对外事件字段必须统一语义类型

对外观测事件 MUST 使用统一的语义类型表达时间点与时长。

#### Scenario: 事件发生时间使用时间点类型

- **WHEN** 某个运行时观测事件被发布
- **THEN** 该事件必须使用 `Instant` 表达事件发生时间

#### Scenario: 耗时与超时使用时长类型

- **WHEN** 某个运行时观测事件需要表达耗时或超时阈值
- **THEN** 该事件必须使用 `Duration` 表达时长

### Requirement: 内核必须在关键图生命周期发布事件

内核 MUST 在图执行关键生命周期发布标准图级事件。

#### Scenario: 图开始时发布开始事件

- **WHEN** 一次图执行刚刚开始
- **THEN** 系统必须发布图开始事件

#### Scenario: 图结束时发布结束事件

- **WHEN** 一次图执行收敛到最终状态
- **THEN** 系统必须发布图结束事件

#### Scenario: 图超时时发布超时事件

- **WHEN** 图执行等待超时
- **THEN** 系统必须发布图超时事件

#### Scenario: 图超时后仍发布结束事件

- **WHEN** 图执行因超时进入最终终态
- **THEN** 系统最终仍必须发布图结束事件
- **AND** 该结束事件的状态必须为 `TIMED_OUT`

### Requirement: 内核必须在关键任务生命周期发布事件

内核 MUST 在任务执行关键生命周期发布标准任务级事件。

#### Scenario: 任务开始时发布开始事件

- **WHEN** 某个任务真正进入执行逻辑
- **THEN** 系统必须发布任务开始事件
- **AND** 该开始事件应能表达该任务在线程池中的排队等待时长

#### Scenario: 任务成功时发布成功事件

- **WHEN** 某个任务成功完成并写入成功状态
- **THEN** 系统必须发布任务成功事件

#### Scenario: 任务失败时发布失败事件

- **WHEN** 某个任务执行失败
- **THEN** 系统必须发布任务失败事件

#### Scenario: 任务超时通过失败事件表达

- **WHEN** 某个任务因超时被取消并进入失败链路
- **THEN** 系统必须发布任务失败事件
- **AND** 该失败原因必须能够通过框架自己的超时异常类型稳定识别

#### Scenario: 任务超时只统计实际执行时间

- **WHEN** 某个任务配置了 `timeoutMillis`
- **THEN** 该超时必须从任务真正进入执行逻辑时开始计时
- **AND** 不得把任务在线程池中的排队等待时间计入该任务超时

#### Scenario: 任务排队时间仅用于观测

- **WHEN** 系统对外暴露任务排队等待时长
- **THEN** 该字段必须仅作为运行时观测数据暴露
- **AND** 不得改变任务成功、失败、跳过与超时的既有调度语义

#### Scenario: 任务被跳过时发布跳过事件

- **WHEN** 某个任务因上游失败而被标记为跳过
- **THEN** 系统必须发布任务跳过事件

#### Scenario: 任务提交失败时发布提交失败事件

- **WHEN** 某个任务提交到底层线程池时被拒绝或提交失败
- **THEN** 系统必须发布任务提交失败事件
- **AND** 该事件必须能够保留底层原始提交失败原因

### Requirement: 观测逻辑不得影响 DAG 主执行流程

运行时观测 MUST 遵守 best-effort 原则，观测逻辑自身异常不得影响 DAG 主执行流程和最终执行语义。

#### Scenario: 单个处理器抛异常不影响主流程

- **WHEN** 某个观测处理器在处理事件时抛出异常
- **THEN** 图执行主流程必须继续进行

#### Scenario: 单个处理器抛异常不影响其他处理器

- **WHEN** 多个观测处理器同时存在且其中一个处理器抛出异常
- **THEN** 其他处理器仍然必须能够继续接收事件

### Requirement: 观测能力不得侵入结构化执行结果

运行时观测 MUST 与业务侧执行结果解耦，不得因首版观测能力引入额外的执行结果膨胀。

#### Scenario: 图执行结果保持极简结构

- **WHEN** 调用方获取 `GraphExecutionResult`
- **THEN** 首版运行时观测能力不得要求把观测事件明细放入执行结果对象
