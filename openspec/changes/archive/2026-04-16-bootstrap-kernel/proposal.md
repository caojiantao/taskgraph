## Why

旧原型仓库已经验证了“基于依赖关系进行并发执行”这条路线的价值，但现有原型把图定义、调度逻辑、Spring 集成和运行时状态耦合在了一起。这会让核心执行语义难以演进、测试，也不利于把它沉淀成一个稳定可用的开源框架。

现在正适合建立一套清晰基线：明确将项目定位为“单机 DAG 任务调度框架”，先沉淀一个小而稳的内核，再在此基础上叠加后续包装能力。

## What Changes

- 使用 OpenSpec 为单机 DAG 调度内核建立变更主线，并以 spec 作为事实来源。
- 引入 kernel-first 架构，拆分图定义、图校验、调度、执行状态与接入层。
- 明确核心执行语义，包括依赖解析、终态、超时与失败处理。
- 定义结构化执行结果模型，供调用方和后续包装层稳定复用。
- 将线程监控、线程池参数动态下发等能力留给后续扩展阶段处理。

## Capabilities

### New Capabilities
- `single-node-dag-kernel`：用于构建、校验和执行单机 DAG 任务图的核心能力，提供确定性的运行时语义与结构化执行结果。

### Modified Capabilities

- 无。

## Impact

- 影响项目整体架构，预计会在新的 `taskgraph` 仓库中建立独立的内核主线。
- 为后续 Java DSL、Spring 支持、监控与运行时控制等能力建立统一的行为基线。
- 通过先锁定内核边界，再进入实现阶段，降低后续演进风险。
